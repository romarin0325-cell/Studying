Card Game Structure Visual Map

`card/` 게임은 전통적인 클래스 계층보다 전역 데이터, 싱글턴 모듈, `RPG` 중심 객체를 조립해 동작합니다. 이 페이지는 유지보수자가 먼저 봐야 할 연결 관계와 흐름을 시각적으로 정리합니다.

모듈 의존 지도

주요 모듈 카드

RPG / index.html

상태와 UI 진입점의 중심입니다. 화면 전환, 덱, 전투 UI, 가챠, TOEIC, Lumi 화면이 모여 있습니다.

statebattlerenderevents

RPGFeatureModules / rpg_features.js

`RPG`에 세이브, 보상, 미션, 모드별 진행 메소드를 주입합니다.

installsavemissionsrewards

BattleRuntime / battle_runtime.js

전투 초기화, 플레이어/적 턴, 스킬 실행, 필드 버프 만료를 처리합니다.

turnsskillsbuffsdeath

Logic / logic.js

스탯, 데미지, 회피, 특성, 적 행동 결정처럼 UI와 분리된 규칙 계산을 담당합니다.

statsdamagetraitsAI

GameUtils / logic.js

카드 조회, 카드 풀 생성, 가중치 추첨, 가챠 등급 계산 등 공통 유틸을 제공합니다.

cardspoolgachashuffle

LumiQuestionRuntime / api.js

Lumi 채팅 세션, 모델 선택, 검색 토글, 취소/재시도, TOEIC 리뷰 컨텍스트를 관리합니다.

sessionmodelcancelTOEIC

핵심 처리 흐름

1

앱 로드

index.html이 데이터와 로직 스크립트를 순차 로딩합니다.

2

모듈 주입

hydrateModules가 RPGFeatureModules를 RPG에 설치합니다.

3

새 게임

startGame과 initNewGame이 저장 데이터와 런 상태를 준비합니다.

4

덱/전투

BattleRuntime이 전투 객체를 만들고 턴을 진행합니다.

5

규칙 계산

Logic과 SideEffects가 데미지, 버프, 특성을 계산합니다.

6

보상/저장

승패 처리 후 미션, 해금, 세이브를 갱신합니다.

장기 유지보수 리스크

  • `RPG`가 UI, 상태, 전투 진입점, 학습 기능을 모두 들고 있어 영향 범위가 큽니다.
  • `Object.assign` 주입 구조는 메소드 출처와 이름 충돌을 정적으로 파악하기 어렵습니다.
  • `LumiQuestionRuntime`에 중복 메소드명이 있어 뒤쪽 정의가 앞쪽 정의를 덮어씁니다.
  • 전역 상수와 스크립트 로딩 순서 의존이 강해 모듈 경계가 약합니다.
  • 전투 규칙이 `Logic`, `BattleRuntime`, `RPGFeatureMethods`에 흩어져 새 카드 추가 시 누락 위험이 있습니다.
